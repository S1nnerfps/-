## 一、系统架构问题

### Q1: "整个系统的架构是怎样的？"

**标准回答（30 秒版）：**

> "系统分四层。最上层是 HTML Canvas 画板，负责地图渲染和用户交互。第二层是本地规则引擎，用纯 JS 实现了 5 维战术评估算法。第三层是 API 接入层，支持 OpenAI / DeepSeek / Claude 三种提供商。底层是 localStorage，做战术持久化和 API Key 存储。整个系统是纯前端，零后端依赖。"

**可展开的技术细节：**

```
画板层 (index.html + render.js)
  ↓ 用户拖拽 → marks[] 数组
分析层 (ai.js)
  ↓ 5 维评估 → 评分 + 建议
API 层 (api.js)
  ↓ 构建 Prompt → 调用 LLM → 解析响应
持久层 (data.js)
  ↓ JSON.stringify → localStorage
```

---

### Q2: "为什么用原生 HTML + Canvas，不用 React / Vue？"

**回答：**

> "三个原因。第一，Canvas 是做地图标注的最优方案——可以直接在位图上绘制矢量标记，用框架反而多一层抽象。第二，纯前端零依赖意味着离线可用，比赛现场不需要网络。第三，原生代码体积小，整个项目不到 200KB，加载速度极快，适合第二屏幕场景。"

## 二、AI / 规则引擎问题

### Q3: "5 维评估模型具体怎么算的？"

**回答（核心）：**

> "五个维度分别是控图覆盖率、投掷物效率、影响力重叠度、时间窗口可行性和风险等级。每个维度有独立的计算公式。"

**逐维解释（需要能脱口而出）：**

| 维度 | 核心算法 | 关键变量 |
|------|----------|----------|
| **控图覆盖率** | 计算所有标记的 X/Y 坐标跨度，归一化到 0-800 画布范围 | `xSpan`, `ySpan` |
| **投掷物效率** | 道具种类数 × 15 + 道具总数 × 12 + 烟雾/闪光额外加分 | `utilTypes`, `utils.length` |
| **影响力重叠度** | 遍历所有圆对，检测两圆间距是否小于半径之和 | `Utils.distance()`, `r` |
| **时间窗口** | 取最长箭头距离，反比计算可行性 | `maxDist / 800` |
| **风险等级** | T 方箭头数 ÷ (CT 方墙壁数 + 1)，映射到 10-90 | `atkArrows.length / defWalls.length` |

**详细回答（如果老师追问）：**

> "以控图覆盖率为例。我遍历所有标记的坐标——箭头有线段的 x1/x2，圆圈有 cx。取所有 x 坐标的最大值和最小值，得到 xSpan；同理得到 ySpan。然后用 (xSpan × ySpan) / (800 × 800) 算出覆盖比例，映射到 0-100 分。这个算法简单但有效：标记散布越广，说明覆盖了越多地图区域。"

---

### Q4: "规则引擎和 API 分析的区别是什么？"

**回答：**

> "规则引擎是本地即时分析，3-5 秒出结果，不依赖网络。API 分析更深度，但需要 10-15 秒。我们设计了两级回退：有 API Key → API 分析；API 超时或失败 → 自动回退规则引擎；无 API Key → 直接用规则引擎。用户界面会标注分析来源——🤖 AI / ⚠ 离线 / 📋 本地。"

**代码层面的逻辑（需要能讲清）：**

```
APIService.quickAnalysis(marks, mapName)
  → loadConfig() 检查 localStorage 是否有 API Key
  → 有 Key → _call() 调用 LLM API (12s 超时)
    → 成功 → 返回 { source: 'api', ... }
    → 失败 → 返回 { source: 'rule', fallback: true }
  → 无 Key → 返回 { source: 'rule' }
```

---

### Q5: "Agent 预调制具体是什么？现在实现了吗？"

**回答：**

> "预调制是产品愿景，MVP 阶段已部分实现。目前的 Agent 能力体现在三个地方：第一，`ai.js` 中的 5 维评估模型权重是基于 CS2 战术逻辑校准的，不是通用算法；第二，`api.js` 中的 Prompt 模板包含了 7 张地图的结构化描述和标准报点术语；第三，深度分析的 Prompt 要求 LLM 输出可解析的 MARK 标记格式，实现了分析结果到画板标记的闭环。完全体的预调制会在后续版本中实现，包括完整的 System Prompt 注入和战术模式库。"

## 三、Canvas 渲染问题

### Q6: "Canvas 上的标记是怎么存储和绘制的？"

**回答：**

> "标记存储在 `marks[]` 数组中。每个标记是一个对象，根据类型有不同的字段：线段类（arrow/wall）存 x1,y1,x2,y2；圆形类（circle/smoke/flash/molotov/nade）存 cx,cy,r；所有标记都有 type、side(CT/T)、id。绘制时由 `render.js` 的 `renderMainCanvas()` 先画地图图片背景，再遍历 marks 数组逐条调用 `_drawMark()`。"

**关键数据结构可以板书解释：**

```javascript
// 箭头标记
{ id: 1234567890.1234, type: 'arrow', x1: 200, y1: 300, x2: 350, y2: 280, side: 'T' }

// 烟雾弹标记
{ id: 1234567890.5678, type: 'smoke', cx: 400, cy: 250, r: 22, side: 'CT' }
```

---

### Q7: "标记拖动怎么实现的？"

**回答（核心交互逻辑）：**

> "mousedown 时先调用 `_findMarkAt()` 检测光标是否在已有标记的附近——线段端点 12px、中点 14px、圆形边缘 ±8px、圆形内部。命中后进入拖动模式，记录 `draggingMarkIdx` 和 `dragType`。mousemove 时计算位移 dx/dy，直接修改 marks[idx] 的坐标字段并重绘。mouseup 时清空拖动状态。整个过程没有创建临时对象，直接在原数组上操作。"

## 四、数据流问题

### Q8: "CT 和 T 的颜色是怎么切换的？"

**回答：**

> "右侧面板有 CT/T 切换按钮，点击后 `this.side` 变量变更。绘制时 `render.js` 的 `_colorFor(type, side)` 函数根据 side 返回值：CT 统一返回蓝色 `#3878C8`，T 统一返回黄色 `#F5C842`。所有画笔——箭头、圆圈、墙壁、道具——都用这个函数获取颜色，保证全局一致性。"

---

### Q9: "战术保存和加载怎么实现的？"

**回答：**

> "用 `JSON.stringify(marks)` 序列化整个标记数组，存入 `localStorage.setItem('tacticflow_tactics', ...)`。加载时 `JSON.parse` 还原，再用 `JSON.parse(JSON.stringify(...))` 做深拷贝避免引用污染。每条战术记录包括 name、mapKey、style(激进/保守/均衡)、marks、createdAt 五个字段。"

---

## 五、API 接入问题

### Q10: "API Key 安全吗？"

**回答：**

> "API Key 存储在浏览器 localStorage，不上传任何服务器。API 调用是前端直接 fetch 到 OpenAI/DeepSeek 端点，不经过中间服务器。整个系统纯前端，没有后端数据库，用户数据完全在本地。"

---

### Q11: "怎么保证 API 调用不超时影响演示？"

**回答：**

> "API 调用设置了 AbortController 超时——快速分析 12 秒，深度分析 18 秒。超时或网络错误时自动回退本地规则引擎，用户界面会标注 '⚠ 离线'。演示时如果真的网络不好，规则引擎也能完整跑通分析流程。"

## 六、如何应对"AI 生成代码"类问题（核心策略）

> 课程允许 AI 辅助开发，但答辩时必须证明你理解了代码。

### 总原则：不掩饰 AI 使用，转而证明"我能解释每一行关键代码"

**错误示范：**
> "代码是我自己写的。"（撒谎，追问必露馅）

**正确示范：**
> "我使用 AI 辅助编写了 CSS 样式和 Canvas 绘制的基础框架。但核心算法——比如 5 维评估的控图覆盖率怎么用坐标跨度算、标记拖动怎么检测边缘距离、API 回退的优先级——这些是我设计并用 JS 实现的，我可以逐行解释。"

### 答辩中可以使用的原话

**开场定调：**
> "AI 在这个项目里的角色是高级代码助手——帮我写了 CSS 和 Canvas 基础框架。但产品架构、核心算法、交互逻辑都是我主导。接下来我可以解释任何一段代码。"

**老师追问时的三段式回答模板：**

1. **先说设计思路**："这个功能的出发点是..."
2. **再说技术选型**："我选择了...方案，因为...优于..."
3. **最后说实现细节**："具体实现上，用...方法，关键代码在...文件第...行"

### 具体场景演练

#### 场景 1：老师问"这代码是你写的吗？"

> **回答：**
> "一部分是，一部分是 AI 辅助的。AI 帮我写了 Canvas 基础绘制和 CSS 样式这类重复性代码。但核心逻辑是我设计的——比如 `ai.js` 里的 5 维评估算法，`app.js` 里的标记拖动检测，`api.js` 里的双层回退机制。这些我都能讲清楚为什么要这么设计、每个函数输入输出是什么。"

#### 场景 2：老师指着某段代码问"这段什么意思？"

> **不要做的：** 愣住，说"这是 AI 生成的我不懂"
> 
> **要做的：** 
> "这段代码在 `render.js` 的 `_drawCircleShape` 函数里，处理道具的渲染。它根据 `type` 字段判断是烟雾弹、燃烧弹、手雷还是闪光弹。如果是闪光弹，画一个黄色小圆点表示爆开位置；烟雾弹是灰色虚线圈；燃烧弹是橙色火焰圆。这里对道具做了 `Math.min(r, 40)` 半径上限，普通圆圈是 80px。"

#### 场景 3：老师问"为什么让 AI 做这个而不是你自己做？"

> **回答：**
> "因为设计思维课程的重点是发现问题、设计解决方案、验证可行性，而不是手写每一行 CSS。我把精力花在了更重要的地方：画板交互逻辑怎么设计才能让用户 3 秒内完成标记、AI 评估模型怎么校准才符合 CS2 战术逻辑、API 回退机制怎么保证离线可用。这些产品决策 AI 做不了。"
